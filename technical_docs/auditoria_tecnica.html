3. Análisis de Seguridad

  • Inyección (SQL): El uso de consultas parametrizadas a través de aiosqlite reduce considerablemente el riesgo de Inyección SQL.
  • Acceso concurrente no autorizado: Al utilizar un archivo local para SQLite, la seguridad depende estrictamente de los permisos del filesystem POSIX (chown/chmod).
  • Exposición de datos en memoria: Los estados globales mantienen tokens, hashes de transferencia y detalles de usuarios directamente en variables de Python en texto claro. En caso de volcado de memoria (Memory Dump), todos los datos en tránsito son expuestos.

4. Análisis de Riesgo Operativo

Riesgo Mitigado ✅: Volatilidad de Estado mitigada con Graceful Shutdown y Checkpointing.
  • Impacto de un Reinicio (Crash/Corte de Energía): La memoria contiene estados críticos de cronómetros (expiración de sesiones). Aunque la información principal está protegida gracias a los backups cada 60s, un fallo de *hardware* del volumen de Fly.io sigue siendo un Punto Único de Fallo.
  • Desincronización de Cola: La cola se sincroniza a disco de manera agrupada cada 3 segundos. Se ha implementado un Graceful Shutdown que fuerza un último volcado antes de matar el proceso, garantizando consistencia si se apaga voluntariamente.

5. Gestión de Recursos

  • Saturación de Memoria (RAM): La carga escala directamente con el número de usuarios manteniéndose en los diccionarios (state.active_connections, state.active_users).
  • Comportamiento ante Exceso: Python no liberará memoria instantáneamente. Si se agota la RAM física del SO, el OOM-Killer detendrá súbitamente el proceso Uvicorn (FastAPI).

6. Estimación de Carga (Load Stress)

Cuello de Botella Solucionado ✅: Se ha eliminado la contención de I/O gracias a la concurrencia asíncrona de Memoria.
  • Escrituras Asíncronas en Memoria: Ahora SQLite ya no se bloquea en cada petición. Las validaciones ocurren en microsegundos contra la memoria RAM, y SQLite se actualiza "tranquilamente" en segundo plano cada 3 segundos sin impactar a los clientes.
  • Lecturas Extremas con WAL Mode: Al activar journal_mode=WAL, cientos de clientes pueden leer asientos y estados a la vez de la base de datos sin que la escritura atómica bloquee el archivo. El servidor puede soportar tranquilamente miles de transacciones lógicas por segundo (TPS en memoria).

7. Mitigaciones Aplicadas (✅ Implementado)

✅ 1. Habilitado WAL Mode en SQLite Se configuró el pragma journal_mode=WAL y synchronous=NORMAL en la inicialización. Permite lecturas simultáneas mientras un hilo realiza escrituras concurrentes de forma ininterrumpida.
✅ 2. Colas de Persistencia y Checkpointing Se eliminó el impacto al disco por cada micro-estado. Las interacciones se manejan en RAM (queue_dirty) y se realiza un *Upsert Batch* hacia el disco duro cada 3 segundos mediante una tarea de fondo (persist_state_task).
✅ 3. Graceful Shutdown Activo Se atrapan las señales SIGTERM y SIGINT. FastAPI pausa el apagado hasta inyectar el último *flush* de estado volátil desde la RAM hacia SQLite, garantizando cero pérdidas de sesión durante un despliegue.
✅ 4. Backups Locales Rotativos Mediante la tarea asíncrona backup_db_task, se clona toda la base de datos cada 60 segundos hacia data/backups/ sin romper transacciones. Además, se purgan automáticamente los archivos con más de 24 horas de antigüedad. (Nota: Sigue existiendo riesgo de pérdida total si falla el disco físico de Fly.io).
